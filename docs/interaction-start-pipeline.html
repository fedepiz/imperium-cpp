interaction_start — the staged composition pipeline

A plan for the rework sketched in src/game.hpp:1619–1669: replace the two branch-per-scope loops with a data pipeline — instantiate scopes once, join defs against instances, splice from a variable table, restore authored order at the end. No code has been changed; this document details the process as intended, flags the traps, and proposes the cheaper cuts.

§ 1Why reshape it

Today interaction_start runs two def loops — texts, then choices — and each loop re-implements scope expansion inline. The Occupants branch walks children_of, builds a TemplateVar array, and emits per element; the unscoped branch builds a nearly identical array and emits once. The same fork appears twice, once per record kind.

Today — scope logic inline, twice

for def : texts
  if Occupants → walk children, vars, emit
  else → vars, emit
for def : choices
  if Occupants → walk children, vars, emit
  else → vars, emit

Intended — scopes instanced once, defs join

instantiate scopes → entries + vars
for def : texts   → join entries, splice
for def : choices → join entries, splice

The costs of the current shape, in order of how much they matter:

  • Every new scope is 2×N branches. Adding a third scope (say, things at the same map cell) means editing both loops. The Scope enum comment already promises "iteration lives in code; data names the set" — but the iteration is copy-pasted per record kind, not owned once.
  • Variables are hard-coded per branch. $TARGET/$SUBJECT arrays are rebuilt per instance, and a new variable ($LOCATION, $KIND) would touch all four branches.
  • Redundant walks. children_of runs once per scoped def, and resolve_name once per def × element. Harmless at current scale, but it is work the shape forces, not work the problem needs.

The intended rework moves scope iteration and variable production into data built once per start — the same move FACT_DEFS already made for facts. That is the right instinct; the rest of this document is about landing it without losing the invariants the current code quietly holds: authored order, occupant order, the instance gate, and replay determinism.

§ 2Already landed in the diff

Parts of the working diff are finished cleanups, independent of the pipeline:

  • trigger_accepts → facts_match — better name, matcher unchanged.
  • facts_sort (bespoke insertion sort) deleted in favor of sort::unstable at both call sites. Fine: fact keys are compared by (kind, value) and duplicates don't care about tie order.
  • Scope::Root added to the enum — semantics still to be decided (P5).
  • #include "list.hpp" added but nothing uses it yet. Note: the staging vectors need to be sorted, and list::List has no random indexing — vec::Vec + Slice is the right container everywhere in this pipeline. If no stage ends up wanting a queue, drop the include.

§ 3The intended pipeline

Six stages. Stage 0 runs at compile/load time (interactions_compile); stages 1–5 run per interaction_start, all staging data on a scratch watermark.

0 · compiledefs carry position, stored sorted by scopeTextDef · ChoiceDef
→
1 · instancescopes → entries {scope, target}Vec<ScopeEntry>
→
2 · varsentry × name → value tableTemplateVar runs
→
3 · joindefs × entries of their scopeinstanced defs
→
4 · ordersort instances by authored position(position, ordinal)
→
5 · emitsplice + gate, build the Interactiondescription · choices

Stage 0 — positions at compile, defs sorted by scope

Each TextDef/ChoiceDef gains a position: its index in file order, captured at vec::push time in interactions_compile — before any re-sort, and per kind (texts and choices order independently; they land in different output lists). Then each slice is sorted by (scope, position):

struct TextDef {
    TriggerDef trigger;
    Scope      scope;
    u32        position; // file order within texts — restores authored order after the scope sort
    String     text;
};

// interactions_compile, after both vecs are built:
sort::unstable(in->texts,   def_less);  // (scope, position) — keys unique, so
sort::unstable(in->choices, def_less);  // instability is harmless and no scratch is needed

Sorting here, once per load, is the right call — sorting per interaction_start would repeat load-time work at runtime. The composite key matters: sorting by scope alone with a heapsort scrambles same-scope defs (P3 is the same trap one stage later).

Stage 1 — instantiate scopes

One place owns "what does each scope expand to": a vector of ScopeEntry {scope, target}. Root contributes the interaction's target; Occupants one entry per children_of(LocationOf, target), in hierarchy order. Built by iterating scopes in enum order, the list is sorted by construction — no sort call, no comparator, and crucially no ordering derived from Id (P2).

Stage 2 — the variable table

Rows of entry × name → value. Reuse TemplateVar as the row payload and lay each entry's rows contiguously, so an entry's variables are one Slice<TemplateVar> — exactly what interpolate takes:

struct ScopeEntry {
    Scope              scope;
    Id                 target;
    Slice<TemplateVar> vars; // this entry's contiguous rows of the var table
};
// per entry: {"TARGET", resolve_name(entry.target)}, {"SUBJECT", subject_name}

$SUBJECT is the same for every entry; duplicating the row per entry (2 × E rows total) is simpler than teaching interpolate a second slice. resolve_name now runs once per entry instead of once per def × element. A new variable becomes a row appended here — no other stage changes.

Stage 3 — join defs against entries

For each def that passes facts_match, pair it with every entry of its scope. With defs sorted by scope and entries grouped by scope, this is a merge join — but see §6-B: a tiny per-scope runs array makes the join an array lookup and needs no cursors. The sketch's block-local template<typename T> InstancedDef does not compile as written (P1); the staged record wants to be a plain POD of indices anyway:

struct Staged {
    u32 position; // the def's file-order index (per kind)
    u32 ordinal;  // the entry's index within its scope run — occupant order
    u32 def;      // index into the (scope-sorted) def slice
    u32 entry;    // index into the entries vec
};

Stage 4 — restore authored order

Sort the staged records by (position, ordinal). Both components are load-bearing: position restores file order across defs, ordinal preserves occupant order among the instances of one scoped def — with a bare position key and a heapsort, "Wanderer / Trader" can come out "Trader / Wanderer" on any given run (P3).

Stage 5 — emit and gate

Walk the sorted staged records and splice directly into the interaction through the existing interaction_add_text / interaction_add_choice, handing each the entry's var slice. Staging spliced strings (TextFragment{String}, ChoiceInstance{Choice}) is the one part of the sketch worth dropping: interpolate writes into a DynString<N>, so staged text would need a new arena-targeting interpolate variant plus a copy per fragment (P4). Sorting references first and splicing last gets the same output with the machinery that already exists.

The gate is unchanged in meaning: staged counts per kind are known before emission — zero text instances or zero choice instances is a content gap, LOG and roll the zeroed interaction back. One caveat: the interaction header must still be written before facts_compute, because fact producers read the situation out of world->interaction (P7).

§ 4Worked example — meeting Town1

Real data from data/interactions.txt. The player meets Town1; facts compute to target_kind = town; suppose two occupants in hierarchy order, Wanderer then a second person, call it Trader. Choices trace (positions are file order among choice records):

posrecordscopefactsinstances (position, ordinal)
0"Greet $TARGET" · meetOccupantstown ✓Greet Wanderer (0,0) · Greet Trader (0,1)
1"Enter" · enterRoottown ✓Enter (1,0)
2"Greet"Rootperson ✗ — dropped—
3"Move on"Root(none) ✓Move on (3,0)

After stage 0 the stored def order is scope-major — Enter (1), Move on (3), Greet $TARGET (0) — so the join emits instances in that order: (1,0) (3,0) (0,0) (0,1). The stage-4 sort restores (0,0) (0,1) (1,0) (3,0), and the final prompt matches today's byte for byte:

You have reached Town1. It is a town. Wanderer is here. Trader is here.
1  Greet Wanderer
2  Greet Trader
3  Enter
4  Move on

Note the tie at position 0: two instances of the same scoped def. That tie is exactly where an unstable sort without the ordinal component reorders occupants at random — and choice order is replay-visible, since resolved choices travel as indices in the command log.

§ 5Pitfalls

P1 · Block-scope template does not compileblocker

template <typename T> struct InstancedDef is declared inside interaction_start — C++ forbids template declarations at block scope; this fails to parse. Options: hoist it to namespace scope next to TextDef, or — better — don't template it. The staged record never needs the def pointer's type: keep per-kind staging vecs of the plain Staged POD above, and let the two emit loops index their own def slice. Two short loops over a shared shape, no generic machinery.

P2 · ScopeEntry's defaulted <=> is deleted — and ordering by Id is wrong anywayblocker

Two independent problems in auto operator<=>(const ScopeEntry&) const = default:

  • Id defines only operator==, so the defaulted <=> is implicitly deleted — it compiles until the first <, then fails where the sort instantiates.
  • Even with <=> added to Id, its order is slot order — spawn order, not hierarchy order. Sorting entries would shuffle occupant lines relative to today's children_of order.

Fix: never sort entries. Instantiate scopes in enum order (stage 1) and the list is grouped by scope by construction, occupants staying in hierarchy order. Drop the operator entirely; nothing needs it.

P3 · Tie-stability in the reorder sortblocker

All instances of one scoped def share that def's position. sort::unstable is a heapsort — it scrambles ties, so occupant instances would reorder arbitrarily between runs. That breaks display order and, worse, replay: choices resolve by index (choice_index in the command log), so nondeterministic order silently diverges replays. Fix either way:

  • composite key (position, ordinal) with sort::unstable — keys unique, no scratch needed (recommended), or
  • sort::stable with the scratch arena, keyed on position alone.

P4 · Staged spliced strings have nowhere to livedesign

TextFragment{String text} implies splicing before sorting, but interpolate only writes into a DynString<N>: staging spliced text means writing a second, arena-targeting interpolate (duplicating its truncation contract), plus copying every Choice (~270 B with its DynString<256>) twice. Invert the last two stages instead: sort the lightweight Staged records, then splice directly into interaction->description / ->choices via the existing add functions. Same output, no new interpolate, no fragment buffers, and the space-joining in interaction_add_text keeps working because emission happens in final order.

P5 · Nil vs Root — two names for the primary targetdesign

Today Nil means "one instance, about the interaction's target" — which is exactly what Root now says. If both survive on compiled defs, the instancer must emit two identical entries and every scope-keyed lookup handles a duplicate. Decide the boundary: scope_compile normalizes empty/nil/root spellings to Scope::Root, and Nil becomes the ZII placeholder that never appears on a compiled def (a zeroed, never-compiled def stays inert — consistent with ZII). Update the enum comment and the data/interactions.txt header comment when this lands.

P6 · Unwired vecs fault on first pushmechanical

The sketch declares vec::Vec<ScopeEntry> instanced_scopes; bare. A ZII vec is valid to read but push on one faults by design ("a null arena is a budgeting bug"). Every staging vec needs vec::make_vec(&game->arena, cap) under the existing ScratchArena watermark — which also means good capacity hints are free: children_of(...).len is known before allocating entries, and staged counts are bounded by defs × entries.

P7 · The write-first / roll-back dance is load-bearinginvariant

The interaction header (active, subject, target, title) is written before facts_compute because fact producers read the situation from the World — fact_produce_target_kind reads world->interaction.target. A tidy-minded rewrite that defers all writes until after the gate breaks fact production. Keep the dance: write the header, compute facts, stage; if either instance count is zero, LOG and world->interaction = {}. The staged pipeline actually improves the gate — counts are known before any text is spliced — and the choices 255-cap overflow can LOG once with exact counts instead of a flag.

P8 · Don't let the interim states reorder outputsequencing

Stage 0 (defs re-sorted by scope) and stage 4 (order restored) cancel each other out. If the compile-time sort lands while the old file-order emit loops still run, every prompt reorders in the interim — corpus stays green (it plays blind) but recorded command streams and eyeballs both notice. The def sort and the reorder must land in the same commit (see §8).

P9 · Small mechanical notesmechanical

  • Scope needs a Count terminator for the per-scope runs array (house enum style already does this elsewhere).
  • Spelling in the sketch: VarInstace, "interaciton", "definitin", "buidl" — and reuse TemplateVar for the row payload rather than a new {key, value} twin.
  • Scoped def over an empty set must still contribute zero instances (empty Occupants run — falls out naturally, but keep the corpus case in mind).
  • Per-kind position counters, not a global one: description and choice list order independently, and per-kind keeps positions dense.

§ 6Sharper cuts — same design, less machinery

Each proposal keeps the intended architecture (scopes instanced once, defs join, vars in a table) and removes one piece of scaffolding.

A · Entries sorted by construction

Instantiate scopes in enum order — Root, then Occupants — and the entries vec is grouped by scope with zero sort calls. Kills the ScopeEntry comparator (and P2 with it). children_of is walked exactly once per start, down from once per scoped def.

B · A runs array instead of a merge join

Record where each scope's entries start and end while building them. The "match defs against scope instances" step becomes an array lookup — no cursors, no join logic to test:

Array<Slice<ScopeEntry>, (usize)Scope::Count> runs = {};
// stage 1 fills entries in enum order and sets runs[s] over each group

for (const TextDef& def : game->interactions.texts) {
    if (!facts_match(def.trigger.required, facts)) continue;
    Slice<ScopeEntry> run = runs[(u32)def.scope];
    for (usize i = 0; i < run.len; i++) { // stage 3+: stage or emit (def, entry, ordinal=i) }
}

C · Sort references, splice last

The fix for P4, restated as a design rule: the only things staged are 16-byte Staged PODs; strings are spliced once, at emit time, into their final DynString destinations through the untouched interaction_add_text/interaction_add_choice. No new interpolate variant, no double-copied Choice, and the sort moves 16-byte records instead of 270-byte ones.

D · The honest question: is the scope sort + reorder needed at all?

With the runs array, a def finds its entries by lookup — it no longer matters whether defs are sorted by scope. Iterating defs in file order over runs[def.scope] emits authored order by construction: no position field, no compile-time sort, no stage-4 reorder, and P3/P8 evaporate. The var table still amortizes per-entry work, so scope-sorted defs buy nothing at runtime unless iteration flips entry-major (it doesn't need to — vars are precomputed) or scope instantiation becomes lazy (instantiate only scopes some surviving def uses — plausible later, not needed now).

recommendation

Land stages 1–2 plus B and C first — that is ~80% of the win (single instancer, var table, both branch forks deleted) with none of the ordering risk. Add position/sort/reorder (stages 0 and 4) only if a concrete need appears: lazy scope instantiation, or def counts large enough that scope-major batching measurably matters. The staged shape is a fine end state; it just shouldn't be the first commit.

E · Keep compile-time work at compile time

If the def sort stays in the plan (D notwithstanding): positions assigned at parse, sort in interactions_compile once per load/hot-reload — never per start. Already the sketch's intent; stated here so it survives implementation.

F · The end state this sets up: SCOPE_DEFS

Once scope instantiation and variable production live in one place each, they are one step from the FACT_DEFS pattern — a table row per scope: {key, instantiate(game, target, out_entries), vars(world, entry, out_vars)}, with scope_compile walking keys and "adding a scope is one row and its hooks." Don't build it for two scopes; build it when the third arrives and the shape has proven itself (rule of three). The pipeline above makes that refactor mechanical when the day comes.

§ 7Cost budget

With N defs (currently 3 texts + 4 choices), K occupants, E = 1 + K entries, I ≤ N · max(1, K) instances — all staging under one ScratchArena watermark on game->arena:

piecesize / costvs today
entriesE × ~24 B (scope, Id, var slice)new, trivial
var table2E × 32 B (TemplateVar rows)replaces per-instance stack arrays
staged recordsI × 16 B, only if stage 4 keptnew; C keeps them index-only
children_of1 walk per startwas 1 per scoped def
resolve_nameE + 1 callswas per def × element
sortscompile O(N log N); start O(I log I) — zero if D takennone today

Everything is noise at this scale — the budget exists to show the reshape costs nothing, not to justify it on speed. The payoff is structural: adding scope #3 or variable #3 touches one stage instead of four branches.

§ 8Implementation order

Four steps, each independently landable with the corpus test green and a manual Town1 prompt identical before and after (./x.sh test, then run and press R to hot-reload data):

  1. Compile side, order-neutral. Scope::Root semantics per P5 (normalize in scope_compile, accept a root spelling), Scope::Count, position fields captured at push. Defs stay in file order; old loops untouched. Behavior identical.
  2. Instancer + var table. Stages 1–2 with cuts A/B: entries built in enum order, runs array, per-entry var slices. Rewrite the two def loops to iterate runs[def.scope] handing entry.vars to the existing add functions — both scope forks and all four TemplateVar rebuild sites delete. Output order unchanged (defs still file-major).
  3. Staging + reorder — only if D decides for it. Def sort by (scope, position) in interactions_compile, Staged records, stage-4 sort by (position, ordinal), emit loop. One commit — interim states reorder prompts (P8).
  4. Prose. Rewrite the interaction_start header comment (it describes the loop shape), the Scope enum comment, and the data/interactions.txt header; delete the sketch block.

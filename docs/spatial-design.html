Spatial design space

Movement execution × spatial indexing for the day-tick strategy sim — every approach we considered, judged against the same seven meeting topologies, with the bookkeeping each one owes. The two choices are coupled: encounter detection is a spatial query made while positions mutate, and every scheme below is really an answer to one question — what does the query see mid-mutation?

The test gauntlet

Every scheme is judged against these topologies. “Fires” means: emit an encounter event with the true pair and the true meeting cell, exactly once. Two findings up front, learned by hand-tracing all of these:

  • T6 (companions) is scheme-independent. “Newly met” is a semantic notion, not a detection notion — every scheme needs a tick-start co-location snapshot or interaction-layer memory. The clean long-term mechanic is parenting: a follower becomes a LocationOf child of the leader — rides for free, takes no steps, triggers nothing.
  • T4 and T5 are the acid tests. Checking against stale positions creates ghost encounters in chases (T5) and misses convergent arrivals (T4). Stale-snapshot schemes need explicit bookkeeping for both; live-index schemes get them free.

Axis 1 — the spatial structure (who is where)

All sizes at the 500×500 cap (250k cells), 32k things, u16 slots. The two serious candidates differ in one property only: snapshot vs live.

S1 — Rebuilt CSR over cells (snapshot)

The HierarchyIndex pattern: version-stamped, counting-sort rebuild where mutations stop. Exact only at rebuild instants — any consumer running during movement sees the past. That staleness is the root of all the patch machinery in the matrix.

Rebuild: 500KB memset + 250k prefix pass + two 32k passes ≈ 0.1–0.3ms. Fits the house derived-state pattern exactly.

S3 — Linked grid (live)

head[cell] + intrusive next[slot] chains, maintained by every position/on-mapness mutation (spawn, despawn, set_parent, the one blessed cell_pos writer). Exact at every instant — mid-movement queries are truth. Slot 0 being the permanent dummy makes 0 a valid ZII “empty” sentinel everywhere.

Price: invariant discipline. A missed maintenance site corrupts silently — the validation pass gains a check (every chained thing's cell_pos matches its chain; chain population == on-map population).

S2 — Sorted (cellkey, slot) array

32k pairs sorted by packed cell key, rebuilt like S1; query = binary-search range. Memory independent of map size — but at the 500×500 cap S1 dominates it (O(1) row lookup vs log n). The fallback if the map cap ever grows huge again.

S4 — Hashtable cell → chain  ·  S5 — No index

S4 buys memory proportional to occupied cells — worth it only when a full head[] table doesn't fit. At this cap it adds hashing and growth-invalidation for no win. S5 (linear scan per query) is live truth with zero maintenance — and O(steps × things) inside encounter detection: hundreds of millions of reads per 30-day batch. Prototype-only.

Memory at the 500×500 cap

The surprise: the two serious candidates cost identical RAM — the difference is purely rebuild-vs-incremental, never size.

S1 rebuilt CSR564 KB
S3 linked grid564 KB
S2 sorted array256 KB
S5 linear scan0

S3 gains +64 KB if a prev[] array is added for O(1) unlink — chains are short, so walk-to-unlink is fine and prev[] can wait.

Axis 2 — the movement executor (when do steps happen, when do checks run)

M1 — Batch end-of-tick

for each mover: walk full budget
rebuild index   // once
sweep co-occupied cells → events

Cheapest possible: the one rebuild the tick does anyway. Phase-through is structural — fast movers cross within the day and are never seen together (T2, T3). Road encounters can only ever be flavor; interception can never be a mechanic. Events fire against final positions: honest state, but for a crossing there is no meeting cell to report at all.

M2 — Lockstep waves

while any budget remains:
  for each mover:
    step 1 cell  // record from→to
  rebuild index
  same-cell + swap pairs → resolve

The most honest simultaneity: within a wave, order genuinely doesn't matter — checks run after the wave against refreshed truth. Chases and races are fair by construction. Owes: the explicit swap check (T3), rebuild × waves × days, and an ordering rule when one thing lands in two same-wave events.

M3 / M4 — Sequential, inline resolution

// M4 — rounds:
while any budget remains:
  for each mover:
    step 1 cell
    query entered cell (live)
    → resolve inline
// M3: drop the outer loop —
// each mover walks all day

The handler runs at the meeting instant, sees the exact meeting cell, and can halt both parties by zeroing budgets — later movers observe the consequences. Detection completeness depends entirely on Axis 1 (see matrix). M3's artifact: early slots move first all day — an equal-speed chase becomes catchable or not by slot order. M4's round-robin interleaving approximates simultaneity and fixes that for ~3 lines of loop.

M5 — Intention / commit

collect all intended steps as data
resolve conflicts symbolically
  (same-target = converge, reversed = swap)
commit all moves
fire derived events

The lockstep-multiplayer formalism: perfect, order-free simultaneity. At speeds > 1 conflict analysis needs path-overlap reasoning across multi-cell intents, and a resolution that halts a mover must retract already-planned intents. Machinery out of proportion to a single-player sim whose interactions want to run inline. Named for completeness; not recommended.

The matrix — where the bookkeeping lives

The central finding of the whole exploration: the bookkeeping is not a property of either axis alone — it's a property of the combination. Snapshot indexes push complexity into the executor; live indexes absorb it.

Why the swap check disappears under sequential + live. The “simultaneous exchange” doesn't exist sequentially: whoever steps first enters a cell the other still occupies, and the live query sees them right there. Under waves the swap exists precisely because both movers moved before anyone looked. The same logic dissolves the crossing, the convergence (the second arriver finds the first, already linked), and the ghost (the fleer is already unlinked). One mechanism, four test cases.

Events — how encounters become consequences

Inline (M3/M4)

Handler runs at the meeting instant: true meeting cell, true world. It can halt both parties, redirect, or ignore; later movers this tick see the outcome. Despawn during resolution is already safe — mark_for_despawn defers death to despawn_flush. The natural home for ROTK-X-style “you bumped into someone on the road”.

Wave-boundary (M2)

Events collected per wave, fired between waves against true post-wave state. One wave of latency between meeting and consequence, plus an ordering rule when one thing appears in two same-wave events (e.g. lower slot of the pair resolves first).

End-of-tick (M1)

Events fire against final state; all meeting context must be carried in the event record — and for crossings there is no context to carry, because the meeting never had a cell.

Re-fire policy (all)

Per-tick pair dedup lives in movement scratch. “We already met yesterday” is either the tick-start co-location snapshot or — better long-term — the interaction system's own memory.

Batched days must be interruptible. tick takes num_days. An encounter whose interaction needs player input must be able to stop the batch: process day-ticks in a loop, return early with “stopped at day k, interaction pending”. Every executor supports this the same way (stop between day-ticks); inline resolution can even stop mid-day — which is exactly the ROTK-X travel-event feel. Nothing on this page forecloses it; worth deciding when the interaction system lands.

Recommendations, ranked

R1 — recommended

M4 × S3 — sequential 1-cell rounds over a live linked grid

The bookkeeping-free column of the matrix. All seven topologies correct by construction, inline resolution reads truth, chases and races stay fair, zero rebuilds, ~600KB. If pursuit is judged a non-mechanic forever, M3 (full budget per mover) is the same design minus the outer round loop — a 3-line difference, switchable at any time.

Price: chain maintenance localized in the few blessed mutators (spawn / despawn_flush / set_parent / the one cell_pos writer) + one validation-pass check. Open sub-decision: the grid lives in World (embedded arrays, maintained by the existing blessed World mutators) or in Game behind mandatory wrappers — World-resident is the lean, since the structure must mutate in lockstep with World's own writes.

R2 — the rebuild-purist alternative

M2 × S1 — lockstep waves over the rebuilt CSR

Choose this to keep the codebase's single derived-state pattern — everything rebuilt, nothing incrementally maintained — with honest simultaneity. Pays the swap check, per-wave rebuilds, and a wave-boundary event-ordering rule. Nothing about it is wrong; it is simply more moving parts in the executor for the same observable behavior as R1.

R3 — the legitimate first milestone

M1 × S1 — batch + end-of-tick sweep

The genuinely simplest thing that works if in-transit road meetings are not a near-term mechanic: towns, arrivals, entering/leaving, and co-located stacks all work; only crossings and swaps (T2/T3) are sacrificed. Because the pathfinding / cache / order / speed layers are identical under every option, upgrading M1 → R1 or R2 later touches only the movement loop and the index module.

Anti-recommendations

  • M5 — machinery out of proportion to a single-player sim whose interactions want to run inline.
  • S4 at this map cap — hashing and growth-invalidation for no win over a 500KB direct table.
  • S2 at this map cap — S1 dominates it.
  • S5 beyond the first prototype.

What swings with the decision — and what doesn't

Index-agnostic and executor-agnostic, already fully designed, buildable now: pathfinding + next-hop cache, move orders (thing destination, cancel-on-fail), speed and budget accumulators, auto-exit, arrival-as-encounter, validation logging.

The decision swings exactly two artifacts: the movement loop inside tick, and the spatial structure plus its maintenance sites (one rebuild call vs a handful of blessed mutators). Both are small; neither leaks into the rest of the design.
